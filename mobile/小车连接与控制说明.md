# 小车连接与控制说明

本文档说明 OpenHarmony 遥控 APP 是如何连接到小车、并下发控制指令的。内容基于源码分析整理，供开发、联调和排障参考。

---

## 一、整体链路

控制指令的流转是一条分层结构：

```
UI 组件(按钮 / 摇杆 / 滑块)
   └─> carApi            门面入口 (CarUtill/CarApi.ets)
        └─> CarEncode    协议编码，生成 ASCII-Hex 帧字符串 (CarUtill/CarEncode.ets)
             └─> TCPClientManager   TCP 单例，负责连接与发送 (tcp/TCPClientManager.ets)
                  └─> @ohos.net.socket   TCP 连接到小车
```

视频链路是**独立**的：不走 TCP 控制通道，而是用 `Web` 组件加载小车上的 HTTP 视频流。

```
VideoComponents ──HTTP──> http://<小车IP>:<视频端口>/index2
```

### 通信方式：控制用 TCP，不用 REST/HTTP

APP 与小车的**控制通信没有使用 REST 接口，也没有使用 HTTP 请求**，而是直接用 **TCP Socket 长连接**。项目中唯一用到 HTTP 的地方是视频流。两条链路对比：

| 链路 | 协议 | 形式 | 代码 |
|------|------|------|------|
| 控制指令 | TCP Socket (`@ohos.net.socket`) | 自定义 ASCII-Hex 帧，无 HTTP 报文头/method/URL | `TCPClientManager.ets` |
| 视频画面 | HTTP | Web 组件加载视频流 URL（仅一个 GET 流，非 REST 接口） | `VideoComponents.ets` |

控制链路的实现要点：

- `socket.constructTCPSocketInstance()` 创建 TCP 套接字，`socket.connect()` 建立到 `<IP>:6000` 的**长连接并保持**。
- `socket.send({ data: message })` 直接在这条连接上写字节流，发送的是自定义帧 `$0110...#`，**没有任何 HTTP 请求行、Header、状态码**。
- 因此控制指令既不是 `GET/POST`，也没有 REST 的资源路径概念，属于"应用层自定义协议 over TCP"。

**为什么这样设计（从代码可推断的原因）：**

1. **高频低延迟**：摇杆 `tiltEvent` 会连续触发 `CtrlCarEncode`，每次都走 HTTP 的握手/请求-响应开销过大，TCP 长连接更适合实时遥控。
2. **单向下发语义**：控制帧是单向指令（停/前/后/速度），不需要 HTTP 的请求-响应模型。
3. **嵌入式端轻量**：小车端多为 MCU/ROS 节点，解析固定格式字节帧远比运行 HTTP Server 轻量。

> 全项目未使用 `@ohos.net.http`（HTTP 请求 API）做控制通信。

## 二、连接参数

APP 需要三个网络参数才能与小车通信，均保存在 OpenHarmony Preferences（首选项）中。

| 参数 | 默认值 | 说明 | Preferences Key |
|------|--------|------|-----------------|
| 小车 IP | `192.168.1.11` | 小车 WiFi 热点/局域网地址 | `KEY_IP` |
| TCP 控制端口 | `6000` | 控制指令通道 | `KEY_TCP_PORT` |
| 视频端口 | `6500` | HTTP 视频流通道 | `KEY_TCP_VIDEO` |

- 首选项文件名：`net_preferences_utils`
- 相关代码：`entry/src/main/ets/utils/PreferencesUtils.ets`
- 应用启动时在 `EntryAbility` 中初始化：`PreferencesUtils.getInstance().init(this.context)`

> 说明：`TCPClientManager` 内部有一组占位默认值 `127.0.0.1:12345`，但连接前总会被实际配置覆盖，不会真正生效。

## 三、使用步骤（用户视角）

1. 确保手机/模拟器与小车处于**同一局域网**（通常连接小车发出的 WiFi 热点）。
2. 打开 APP，进入「连接」标签页（启动页 `MainConsole` 中的连接页 `fireguard/screens/ScreenConnect.ets`）。
3. 填写小车 **IP**、**TCP 端口**、**视频端口**（首次进入会自动回填上次保存的值，否则用默认值）。
4. 点击「连接」：
   - 成功 → 提示"连接成功"，自动跳转到主控制页；
   - 失败 → 提示"连接失败，请检查信息"。
5. 进入控制页后即可用按钮 / 摇杆 / 滑块操控小车，视频区显示实时画面。

## 四、连接建立流程（技术视角）

连接的核心逻辑在 `ScreenConnect.ets` 的 `onConnect()` → `initTcpConfig()`：

1. `saveNetConfig()`：把界面上的 IP/端口写入 Preferences。
2. 设置视频参数：`VideoComponents.ip` / `VideoComponents.port`。
3. `TCPClientManager.getInstance().initNetAddress({ address, port })`：注入目标地址。
4. `await TCPClientManager.getInstance().connect()`：建立 TCP 连接。

`TCPClientManager.connect()` 的关键行为（`tcp/TCPClientManager.ets`）：

- 若已连接，先 `socket.close()` 断开旧连接。
- `socket.bind({ address: '0.0.0.0', family: 1 })` 绑定本机。
- `socket.connect(options)` 连接小车。
- 注册 `message` 回调用于接收下行数据。
- **连接失败必须 `socket.close()`**（源码注释：在 Mate50 HarmonyOS 4.2 API 9 上，连接失败若不关闭会导致下次无法连接）。

### 连接保持与重连

- **没有主动心跳保活**。`TCPClientSendUtils` 里有一个 5 秒定时器，但心跳发送逻辑已被注释，实际是空转。
- 采用**惰性重连**：每次 `sendMessage()` 发送前先 `isConnect()` 检测状态，若已断开则调用 `connect()` 重连（注意：触发重连的**那一帧数据不会补发**）。
- 状态检测 `isConnect()` 依据 `socket.getState()` 判断 `isClose == false && isConnected == true`。

### 接收数据

- `connect()` 注册了 `message` 回调，会把收到的字节拼成字符串。
- 但最终处理函数 `_receiveMessage` 默认是空实现，`TCPClientReceiveUtils` 的接收处理也被注释。
- **结论：当前 APP 只发不收，下行数据未被业务处理。**

## 五、指令协议（帧格式）

所有控制指令都由 `CarEncode.BaseEncode()` 生成，是一段**以 ASCII 字符表示的十六进制文本帧**：

```
$ + [车辆类型] + [命令标记] + [数据长度] + [数据主体] + [校验和] + #
```

| 字段 | 长度 | 说明 |
|------|------|------|
| 起始符 | 1 | 固定 `$` |
| 车辆类型 | 2 | 固定 `01` |
| 命令标记 | 2 | 见下表命令码 |
| 数据长度 | 2 | = 数据主体字符数 + 2（含校验字节），两位十六进制大写 |
| 数据主体 | 变长 | 各命令载荷，每参数用两位十六进制表示 |
| 校验和 | 2 | 对前面 `code` 逐字节（每 2 字符）累加后 `% 256` |
| 结束符 | 1 | 固定 `#` |

其他约定：

- **负数编码**：负速度值 `+256` 转为无符号字节（等价 8 位补码）。例如 `-1` → `255` → `FF`。
- **十六进制**：`numberToHex` 补零到指定长度并转大写。
- 相关代码：`entry/src/main/ets/CarUtill/CarEncode.ets`。

### 方向枚举（CarDirection）

按钮方向控制（命令 `15`）的数据主体取自该枚举（`CarUtill/CarEnum.ets`）：

| 值 | 含义 | 值 | 含义 |
|:--:|------|:--:|------|
| 0 | 停车 Stop | 4 | 右平移 Right |
| 1 | 前进 Front | 5 | 左旋转 LeftRotate |
| 2 | 后退 After | 6 | 右旋转 RightRotate |
| 3 | 左平移 Left | 7 | 刹车 Brake |

## 七、帧编码示例

以「前进」按钮为例（`CarDirection.Front = 1`）：

1. `ButtonCarEncode(1)` → `BaseEncode('15', '01')`
2. 数据主体 `info = '01'`，长度 2
3. 数据长度 `size = numberToHex(2 + 2, 2) = '04'`
4. 拼接（校验前）`code = '01' + '15' + '04' + '01' = '01150401'`
5. 校验和：`0x01 + 0x15 + 0x04 + 0x01 = 0x1B`
6. 追加校验 → `011504011B`
7. 加起止符 → **最终帧 `$011504011B#`**

> 该示例按源码逻辑推演，实际字节以运行日志（调试模式 `console.log('BaseEncode', code)`）为准。

## 八、UI 如何触发指令

- **方向按钮**（`components/CarBtnComponents.ets`）：3×3 宫格，"按住才动"点动控制。按下发送对应方向，松手统一发 `Stop(0)`。
- **摇杆**（`components/CarRockerComponents.ets` + `Rocker` 库）：手指位置线性映射到 `[-100, 100]` 的 x/y 分量（Y 轴向上为正），通过 `CtrlCarEncode`（命令 `10`）连续下发；松手归位到中心即停车。方向和力度合并在 x/y 数值里。
- **麦克纳姆轮四轮控制**（`pages/MecanumWheel.ets`）：4 个滑块（范围 -100~100）绑定 L1/L2/R1/R2，点"更新速度"经 `UpSpeedCarEncode`（命令 `21`）下发。
- **视频区**（`components/VideoComponents.ets`）：拍照 `60`、开始录像 `61`、结束录像 `62`。
- **循迹**（`pages/RemoteControl.ets`）：自动循迹复选框触发 `63` / `64`。

## 九、排障建议

| 现象 | 可能原因 | 排查方向 |
|------|----------|----------|
| 连接失败 | IP/端口错误、不在同一网段 | 核对小车 IP 与端口；确认已连小车 WiFi |
| 连接成功但小车不动 | 端口对但协议/校验不匹配；小车侧未监听 | 抓包看帧内容；开启调试模式看 `BaseEncode` 日志 |
| 视频不显示 | 视频端口错误、视频服务未启动 | 浏览器直接访问 `http://<IP>:<视频端口>/index2` |
| 断线后指令丢失 | 惰性重连当帧不补发 | 重连后重新下发指令 |
| 无下行反馈 | 接收处理逻辑被注释 | 属已知设计，当前只发不收 |
