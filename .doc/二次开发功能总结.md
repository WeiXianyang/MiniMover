## 三、核心算法与智能模块（⭐ 答辩重点）

### 3.1 烟火检测三级复核体系

**问题背景**：机器人需要在巡检过程中实时检测火灾隐患，但单一 YOLO 模型的误报率无法满足工业场景可靠性要求。

**技术方案**：设计"**本地检测 → AI 云端复核 → 云端去重上报**"三级流水线。

```
摄像头帧流 → YOLOv5(本地) → 滑动窗口触发 → GPT多模态复核 → 云端告警
              ↓                   ↓                ↓              ↓
         置信度≥0.70        2秒内≥5帧命中    base64图片+提示词   幂等写入MySQL
```

**关键设计点**：

1. **滑动窗口触发机制**（`event_manager.py`）
   - 维护一个时间戳双端队列，窗口宽度 2 秒，最小命中帧数 5 帧
   - 仅在滑动窗口内连续检测到足够帧数时触发事件——有效过滤瞬时光照/反光干扰
   - 10 秒无检测后自动清除事件状态，避免持续无效告警

2. **AI 云端复核**（`ai_reviewer.py`）
   - 独立后台守护线程运行，使用单槽队列（maxsize=1）确保同一时间只有一次复核在进行
   - 将原始帧（无标注）JPEG 编码为 base64，发送到 OpenAI 兼容的 GPT 多模态 API
   - 中文提示词精心设计三分类：`confirmed_fire` / `suspected_smoke` / `no_fire`
   - 内置重试机制（最多 3 次）+ 标记代码块清洗 + 置信度范围校验
   - YOLO 推理线程不受 AI 复核阻塞，保证实时检测不中断

3. **告警状态机**
   ```
   IDLE → AI_REVIEWING → ALARMED_FIRE  (confirmed_fire)
                        → ALARMED_SMOKE (suspected_smoke)
                        → AI_REJECTED   (no_fire, 每30秒重新复核)
                        → AI_FAILED     (API异常)
   ```
   - 每个状态转换都有明确的触发条件和处理逻辑
   - AI_REJECTED 状态下，只要本地检测持续，每 30 秒自动重试复核
   - ALARMED 状态下，每 60 秒周期性保存证据快照

4. **证据留存与云同步**（`evidence_store.py` + `cloud_service.py`）
   - 本地证据存储：环形缓冲区，最多 10 张图片，超限按时间戳自动淘汰
   - 原子写入：先写 `.tmp` 临时文件，成功后再 `replace` 替换
   - JSON 元数据侧载文件：记录事件 ID、AI 复核状态、检测类别等
   - 云端上报：两步式（先上传证据图片获取 URL → 后 POST 告警 JSON）
   - **发件箱重试模式**：上传失败不入库，持久化到本地 `cloud_outbox.jsonl`，后台线程指数退避重试（起始 2s，上限 300s）；重启时自动重试积压消息

5. **云端去重**：MySQL 的 `UNIQUE KEY (event_id, alarm_type)` + `INSERT ON DUPLICATE KEY UPDATE` 实现幂等写入

### 3.2 语音助手端到端流水线

**问题背景**：实现离线和在线混合的中文语音交互，支持唤醒词激活和声纹身份验证。

**技术方案**：

**A. ASR 语音识别**（`asr_backend.py`）

采用 **FunASR Paraformer-zh-streaming** 流式模型（ModelScope 自动下载），配合双 VAD 方案：

```
麦克风输入(16kHz/mono)
    ↓
┌── 外部 VAD：Silero VAD (ONNX) ── 判断语音起止
├── 内部 VAD：RMS 能量 + SNR 门限 ── 双层判决
└── 预滚缓冲：400ms 防止语音开头被截断
    ↓
FunASR Paraformer 在线流式解码（lookback=4 encoder chunks）
    ↓
静音间隙 800ms → 触发判句完成
```

- 离线优先，本地运行，无需云端依赖
- 备选方案：OpenAI Whisper API（网络后端），用于远程语音控制场景
- 备选方案：车载麦克风远程采集 + Whisper 识别（多车编队模式）

**B. TTS 语音合成**（`icar_audio.py`）

- **主方案**：阿里云 DashScope CosyVoice v3（`cosyvoice-v3-flash`），自定义语音 `longanyang`
- **备用方案**：开源 TTS API + `espeak-ng` 本地多语言朗读
- 配置通过 `.tts.env` 文件加载，自动从车载 USB 音箱播放

**C. 唤醒词服务**（`wake_word.py`）

设计 IDLE/AWAKE 双状态机：

```
IDLE 状态：全量 ASR 结果 → 子串匹配唤醒词 → 播报问候 → 切换到 AWAKE
AWAKE 状态：全部识别结果 → 转发 VoiceService 处理 → 30秒无交互 → 回退 IDLE
```

- 唤醒词和指令可在同一句话中识别（如"你好小南，往前走"）
- 唤醒词通过 `MINIMOVER_WAKE_WORD` 环境变量配置
- 问候语自动生成，也支持自定义

**D. 声纹验证**（`speaker_verifier.py`）

- 基于 **CAM++** 说话人嵌入模型（FunASR/ModelScope，约 50MB）
- 注册：录制 15 秒语音 → 提取 256 维说话人向量 → 保存 `.npy` 文件
- 验证：实时语音 → 提取嵌入向量 → 余弦相似度 ≥ 0.38 阈值 → 通过/拒绝
- **非停止类运动指令（前进/后退/转弯等）需要声纹验证通过才执行，停止类指令无需验证**
- 这是一个安全机制：防止非授权人员通过语音控制机器人

### 3.3 多车编队协调器

**问题背景**：多台巡检机器人需要协同工作，支持编队控制、碰撞预警、统一监控。

**技术方案**：中心化协调服务器（Flask，端口 8888），管理多车注册、状态采集、编队控制和碰撞检测。

**A. 并行状态采集与碰撞检测**

- 后台守护线程每 5 秒通过 `ThreadPoolExecutor` 并行轮询所有注册车辆的 `/api/status`
- 非阻塞轮询锁（`_poll_lock`）：防止上一轮轮询未完成时新轮次堆积
- 碰撞检测：每轮采集后计算两两车辆间欧几里德距离
  - **关键距离 0.5m** → CRITICAL 红色告警
  - **警告距离 1.0m** → WARNING 黄色告警
  - 告警以 `{car_a}_{car_b}` 为 Key 去重，状态变化时更新

**B. 队形控制（3 种编队模式）**

| 队形 | 布局规则 |
|------|----------|
| **纵队（Line）** | 沿 X 轴间隔 `spacing` 米直线排列 |
| **横排（Row）** | 沿 Y 轴居中排列 |
| **三角（Triangle）** | 首车在前，其余车交替左右两翼，错行排列 |

- 队形目标位置通过每辆车的 `/api/navigate` 端点下达
- 位置计算纯数学运算，无外部依赖

**C. 视频流代理**

- MJPEG 流透明代理：`/proxy/camera/<car_id>` → 车端 `/video_feed`
- 连续 3 次连接失败后生成 OpenCV 占位图（"car_id offline"），优雅降级

**D. 语音服务生命周期管理**

- 每辆车可独立启动语音助手子进程（`subprocess.Popen`）
- 车辆注销时自动 `terminate() → kill()` 清理对应语音进程

### 3.4 交通信号识别（双路异构检测）

针对交通灯检测，实现了两套独立方案：

| 方案 | 算法 | 适用场景 |
|------|------|----------|
| **HSV 色彩空间** | 高斯模糊 → HoughCircles 圆检测 → 遮罩颜色比判定 | 边缘设备快速推理 |
| **YOLOv3 深度学习** | YOLOv3-SPP 模型（6 类），250MB 权重 | PC 端高精度检测 |

**HSV 方案关键细节**：
- 只处理图像上 1/2 区域（ROI 裁切）—— 交通灯通常在画面上方
- 红色处理 HSV 环绕（`0-10°` + `170-180°` 两段合并）
- 颜色比例阈值 0.6（圆形遮罩内目标颜色像素占比）

**HyperLPR 车牌识别**：包含完整的 NumPy / OpenCV / Keras 兼容层，解决旧版库在新版 Python 环境的 API 变更问题

---

### 3.5 人脸识别集成

基于 Django + Baidu Cloud Face API v3，实现双模式身份识别：

- **1:1 验证**：登录时采集面部 → 与 AES-256 加密存储的注册照逐一比对
- **1:N 识别**：单张面部 → 云端多人脸库搜索
- **加密存储**：邮箱、电话、面部均 AES-256 加密（ECB 模式文本 / EAX 模式图片）
- **双摄像头支持**：本地 USB 摄像头 + 远程小车 ROS 视频流

## 四、工程架构亮点（⭐ 答辩重点）

### 4.1 双通道车控架构（低延迟创新）

**问题**：HTTP REST 控车存在延迟瓶颈（TCP 三次握手 + Flask 请求解析 + JSON 反序列化 ≈ 10-50ms/次）。

**方案**：在 HTTP :5000 基础上新增 TCP :5001 控制通道，共享底层运动执行引擎：

```
HTTP 客户端 ──POST /api/move──→ Flask ──→ _execute_move() ──┐
                                                              ├──→ bot.set_car_motion()
TCP 客户端  ──"forward 50 0.5"─→ Raw Socket ──→ _execute_move() ─┘
```

- 两者共享同一个 `_move_lock` 互斥锁，确保串口写入线程安全
- 自动停止定时器防止断连后小车失控
- 支持即时停止：`duration=0` 立即取消前一次定时器 + 刹车

---

### 4.2 零模板嵌入式 Dashboard

多车协调器和车端导航页面均采用**零外部模板**设计：

- 完整 HTML/CSS/JS 内联在 Python 字符串中（无 `.html` 模板文件）
- 单文件即可运行，部署极其简洁
- 自适应轮询频率（状态过渡期 1.5s / 空闲期 5s）

---

### 4.3 Docker 代理的 ROS2 桥接

车端 API 服务器（纯 Python Flask）**不安装 ROS2**，通过 Docker 容器执行 ROS2 命令：

```python
# 所有 ROS2 操作通过 Docker exec 代理
docker exec <container> bash -c "source /opt/ros/foxy/setup.bash && ros2 service call ..."
```

**关键技术**：
- 超时缓存：`container_running()` 和 `patrol_services_ready()` 2 秒 TTL 缓存，防止 Docker exec 风暴
- 协作缓存：`ros_bridge.py` 和 `stack_manager.py` 共享状态缓存，避免重复执行 `ros2 service list`
- 导航栈异步生命周期：后台线程管理启动/停止，Flask 请求立即返回不阻塞
- 初始位姿发布：`PublishWithCovarianceStamped` 消息通过 `subprocess.Popen` 非阻塞发布，最多重试 5 次

### 4.4 二进制传感器协议解析

iCar 传感器驱动仅 81 行代码，解析自定义 6 字节二进制协议：

```
[Header:0xA5][NodeID][Type][Value_L][Value_H][Checksum]
```

- 数据通过 `/dev/ttyUSB2` 串口 (115200bps)
- 含校验和验证，错误数据自动丢弃
- 9 种传感器类型（温度/湿度/烟雾/PM2.5/气压/光照/GPS经纬/CO2）
- 线程安全字典缓存，守护线程后台轮询

### 4.5 云平台全栈设计

**后端**（Flask 3.0.3 + PyMySQL + gunicorn）：

| 特性 | 实现 |
|------|------|
| 告警去重 | `INSERT ON DUPLICATE KEY UPDATE`（幂等） |
| 鉴权 | Bearer Token（可选，开发模式绕过） |
| 权限 | 均需 Token 访问（healthz 除外） |
| 24 小时趋势 | `GROUP BY HOUR()` + Python 补零填充 |
| 车辆状态 | `MAX(received_at) GROUP BY car_id` + 2 分钟在线阈值 |

**前端**（Vite 6 + React 18 + TypeScript + Tailwind CSS + Recharts）：

- 暗色主题工业风，完整响应式
- 自动降级：API 不可用时切换到本地 Mock 数据
- 30 秒自动刷新 + 数字脉冲动画
- 告警表格排序/翻页/详情预览面板
- 24 小时趋势柱状图（Recharts BarChart 渐变填充）
- 车辆在线/离线状态（距离上次心跳的时间差）

### 4.6 模型分发与版本管理

通过 `recognition_models.json` 清单文件统一管理所有 AI 模型：

- SHA256 校验和防篡改
- Git LFS 追踪（YOLOv5 fire/smoke 模型）
- 外部仓库克隆 + Google Drive 下载（YOLOv3 交通灯、HyperLPR 车牌）
- `prepare_recognition_models.py` 一键准备所有模型
- LFS 指针文件自动检测，防止推送占位符

---

### 4.7 CI/CD 与部署

- **车端 CI**：`.github/workflows/car-ci.yml`，编译 `compileall` 全量 Python 语法校验 + Docker 构建，push main 后推送到 GHCR
- **启动脚本**：`start_services.sh` 一站式启动整套车端服务栈（USB 摄像头、ROS2 Docker 容器、web_video_server、API 服务器、烟火检测等）
- **自动设备检测**：脚本自适应检测 Jetson 容器 ID（通过 Docker 镜像名匹配，不写死）
- **多车部署**：`deploy_to_cars.sh` 批量 SSH + SCP 部署到多台小车
