# FireGuard 多模态身份认证与巡检联动设计

## 1. 文档目的

本文将已完成或在建的语音能力，与烟雾火焰、红绿灯、车牌、人脸四类视觉识别能力串联为一条可演示、可拆分开发的业务闭环。设计场景为园区配电房与仓储通道巡检：小车默认静默待机，经授权人员唤醒并完成声纹、人脸双因子认证后进入可控状态；巡检期间四类视觉识别按任务上下文参与导航、通行核验、风险告警与报告归档。

本设计不改变底层底盘控制协议，统一由 `inspection_mission_manager` 和新增的 `inspection_multimodal_orchestrator` 编排既有模块。

## 2. 已有能力与集成目标

### 2.1 当前实现状态

| 能力 | 当前状态 | 本方案中的作用 |
|---|---|---|
| FunASR 2-pass | 已搬运 | 默认 ASR 引擎：流式首结果用于快速反馈，二次结果用于执行指令 |
| Whisper 手动切换 | 保留 | FunASR 不可用或识别置信度低时的人工切换备份 |
| 停止/急停 | 已完成 | 最高优先级安全动作，绕过认证和任务状态 |
| TTS 队列与取消 | 已完成 | 身份认证、任务反馈、告警播报；高等级告警可抢占普通播报 |
| 声纹录入 | 已完成 | 建立授权用户的 `voiceprint_id` 与角色映射 |
| 声纹动作门禁 | 已完成 | 作为唤醒后的第一道身份校验和语音指令门禁 |
| ALSA 设备探测 | TODO | 启动时选择可用麦克风/扬声器设备 |
| scipy 重采样 | TODO | 统一输入音频采样率，避免声纹/ASR 特征不一致 |
| 实机声纹阈值标定 | 待实测 | 决定声纹匹配阈值与误拒/误接收表现 |

### 2.2 集成目标

1. 将“语音控制”提升为“先认证、后授权、可追溯”的控制入口。
2. 将四个视觉模块分配到明确的巡检上下文，而不是并行展示识别框。
3. 将识别事件统一进入任务状态机，形成停止、播报、抓拍、上报、恢复或人工接管的一致流程。
4. 保留 APP 与物理急停作为独立安全通道，避免声纹、人脸或网络故障阻塞紧急制动。

## 3. 核心业务闭环

### 3.1 静默待机到认证成功

1. 小车处于 `SILENT_STANDBY`：不执行导航，不播放普通提示音，仅保留本地唤醒词检测、急停监听和火焰/烟雾安全检测。
2. 用户说出唤醒词，例如“火卫，唤醒”。系统播报“请说出身份确认口令”。
3. `voiceprint_gate` 对说话人声纹进行比对。未通过时只播报“身份未验证”，不暴露候选人员姓名，也不开放控制指令。
4. 声纹通过后，小车停止移动并切换前置相机至人脸认证模式，播报“声纹验证通过，请面向摄像头进行人脸验证”。
5. `face_auth` 在限定时间内识别人脸，并校验人脸所属用户是否与 `voiceprint_id` 映射的用户一致。
6. 双因子认证成功后创建短时 `operator_session`，播报“你好，{姓名}。已进入巡检控制模式”。此时开放与该角色相匹配的语音命令与 APP 控制权限。
7. 任一认证步骤超时或失败均回到 `SILENT_STANDBY`。连续失败达到阈值时记录安全事件，并要求 APP 或管理员重新解锁。

### 3.2 巡检中的四视觉联动

| 视觉模块 | 触发位置/时机 | 业务含义 | 系统动作 |
|---|---|---|---|
| 烟雾火焰识别 | 全程低频检测；配电柜、消防点等高风险点提升帧率 | 火情初筛 | 基于视觉置信度、连续帧稳定性和区域风险进行分级；高等级时停车、抓拍录像、声光告警、TTS 抢占播报、APP 推送 |
| 红绿灯识别 | 仅在地图配置的交叉口、门禁灯或模拟路口区域启用 | 通行许可 | 红灯停止等待，黄灯减速并等待，绿灯或通行灯放行；超时转人工确认 |
| 车牌识别 | 车辆出入口、仓储通道检查点或指定配送点 | 车辆/通行记录核验 | 写入时间、位置、车牌、截图；命中白名单正常记录，未命中或黑名单触发待确认事件 |
| 人脸识别 | 唤醒认证、人工接管前、指定安检点 | 人员身份确认 | 与声纹做双因子绑定；认证成功后语音问候；不用于巡检途中自动跟随或非必要的持续人脸采集 |

### 3.3 典型演示脚本

1. 小车静默停在充电点，屏幕/APP 显示“待唤醒”。
2. 授权人员说“火卫，唤醒”，系统完成声纹比对，提示进行人脸认证。
3. 人脸识别成功，播报“你好，张三。已进入巡检控制模式”。
4. 张三说“开始 A 路线巡检”。系统解析为 `start_mission(route=A)`，导航开始。
5. 到达模拟交叉口，红绿灯识别到红灯，小车停止并播报“前方红灯，等待通行”。识别到绿灯后继续前进。
6. 到达车辆检查点，识别车牌并写入本次巡检记录；若为非授权车辆，播报“发现待确认车辆，已上报”。
7. 在配电柜 A 处发现烟雾/火焰，小车急停，取消普通 TTS 队列，播报“警报：配电柜 A 发现疑似火情，已停止巡检并上报”。同时抓拍、录像、亮灯鸣笛、APP 推送。
8. 授权人员可说“确认告警”或在 APP 点击“人工接管”。处理后选择“继续巡检”“返回起点”或“执行应急配送”。

## 4. 状态机与权限设计

### 4.1 顶层状态

```text
SILENT_STANDBY
  -> WAKE_LISTENING
  -> VOICEPRINT_VERIFY
  -> FACE_VERIFY
  -> AUTHORIZED_IDLE
  -> MISSION_RUNNING
  -> EVENT_HOLD / MANUAL_TAKEOVER
  -> AUTHORIZED_IDLE 或 RETURNING
  -> SILENT_STANDBY

任意状态 --急停/火情高危--> SAFETY_STOP
SAFETY_STOP --人工确认--> EVENT_HOLD 或 AUTHORIZED_IDLE
```

### 4.2 状态转换规则

| 当前状态 | 事件 | 下一状态 | 约束/动作 |
|---|---|---|---|
| `SILENT_STANDBY` | 唤醒词命中 | `VOICEPRINT_VERIFY` | 停留在原地；开始录音与声纹比对 |
| `VOICEPRINT_VERIFY` | 声纹通过 | `FACE_VERIFY` | 发布候选用户 ID；提示面向摄像头 |
| `VOICEPRINT_VERIFY` | 失败/超时 | `SILENT_STANDBY` | 记录失败次数，禁止执行控制命令 |
| `FACE_VERIFY` | 人脸与声纹同一用户 | `AUTHORIZED_IDLE` | 创建会话并 TTS 问候 |
| `FACE_VERIFY` | 失败/超时 | `SILENT_STANDBY` | 清空候选用户与临时人脸帧 |
| `AUTHORIZED_IDLE` | 开始巡检/配送 | `MISSION_RUNNING` | 校验用户角色、地图、设备状态 |
| `MISSION_RUNNING` | 普通识别事件 | `MISSION_RUNNING` | 写入事件记录，不中断任务 |
| `MISSION_RUNNING` | 通行受限/待确认 | `EVENT_HOLD` | 停车、请求语音或 APP 确认 |
| `MISSION_RUNNING` | 高等级火情/急停 | `SAFETY_STOP` | 底盘置零、取消普通播报、告警上报 |
| `EVENT_HOLD` | 继续/返航/人工接管 | 对应状态 | 命令必须来自有效会话、APP 或物理控制器 |
| 任意可控状态 | 会话超时/退出 | `SILENT_STANDBY` | 取消权限；运行中的任务先安全暂停 |

### 4.3 权限矩阵

| 角色 | 声纹+人脸认证后可执行 | 不可执行 |
|---|---|---|
| 巡检员 | 开始/暂停/恢复巡检、查询状态、确认低风险事件、返回起点 | 修改白名单、修改阈值、解除高危告警 |
| 管理员 | 巡检员权限 + 手动接管、应急配送、解除待确认事件 | 修改原始识别证据 |
| 系统维护员 | 管理员权限 + 声纹/人脸录入、阈值配置、设备自检 | 删除审计日志 |
| 任意人员 | “停止”“急停”、物理急停 | 启动、恢复、导航、配送 |

安全原则：急停不需要身份认证；启动、恢复、手动驾驶和敏感配置必须有有效身份。发生高危火情时，即便当前会话有效，也只能进入“确认/接管/返航”流程，不能用普通语音命令绕过告警。

## 5. 模块架构与职责

### 5.1 新增编排模块

新增 `inspection_multimodal_orchestrator`，作为身份认证和视觉事件的策略层；`inspection_mission_manager` 继续作为导航/任务生命周期的唯一总控。

| 模块 | 输入 | 输出 | 责任 |
|---|---|---|---|
| `inspection_voice_agent` | 麦克风音频、TTS 播报请求、会话状态 | ASR 文本、意图、声纹结果、TTS 状态 | 唤醒、ASR、声纹门禁、播报队列 |
| `inspection_face_auth` | 相机帧、候选用户 ID | 人脸匹配结果/置信度 | 限时的人脸认证与人员映射校验 |
| `inspection_vision_detector` | 相机帧、地图区域上下文 | 火情/红绿灯/车牌视觉事件 | 四类视觉模型的统一事件输出 |
| `inspection_visual_fire_judge` | 火焰/烟雾检测结果、地图区域上下文 | 火情等级、证据请求 | 依据置信度、连续帧命中、区域风险和人工确认进行分级 |
| `inspection_multimodal_orchestrator` | 语音、声纹、人脸、视觉、任务状态 | 会话状态、授权命令、事件处置指令 | 双因子认证、权限校验、视觉策略、TTS 优先级 |
| `inspection_mission_manager` | 已授权任务命令、处置指令、导航反馈 | 任务状态、导航/停车/返航动作 | 任务状态机和底盘导航编排 |
| `inspection_event_server` | 会话、视觉、告警、任务事件 | WebSocket/JSON 上报 | APP 推送、审计记录、报告数据 |

### 5.2 建议 ROS2 接口

| 接口 | 类型 | 关键字段 | 说明 |
|---|---|---|---|
| `/voice/wake_event` | topic | `wake_word, timestamp` | 唤醒词检测成功 |
| `/voice/voiceprint_result` | topic | `user_id, score, passed` | 声纹门禁结果 |
| `/face/auth_result` | topic | `user_id, score, passed, image_ref` | 人脸认证结果 |
| `/vision/event` | topic | `event_id, type, confidence, location, image_ref` | `fire_smoke`、`traffic_light`、`plate`、`face` 统一事件 |
| `/multimodal/session` | topic | `session_id, user_id, role, state, expires_at` | 有效用户会话，不发送原始生物特征 |
| `/multimodal/command` | topic | `intent, session_id, slots, source` | 权限校验后的任务命令 |
| `/mission/control` | service/action | `start/pause/resume/return/stop` | 任务编排控制入口 |
| `/safety/emergency_stop` | topic/service | `source, reason` | 最高优先级制动接口 |
| `/event/report` | topic | `severity, evidence_refs, action, status` | 向 APP、报告和云端统一上报 |

### 5.3 统一事件模型

所有视觉和认证事件采用同一事件结构，便于 APP、日志和巡检报告复用：

```json
{
  "event_id": "evt-20260713-0001",
  "type": "fire_smoke",
  "severity": "HIGH",
  "timestamp": "2026-07-13T10:30:12+08:00",
  "mission_id": "mission-A-01",
  "location": "配电柜A",
  "confidence": 0.93,
  "evidence_refs": ["capture/evt-0001.jpg", "capture/evt-0001.mp4"],
  "action": "safety_stop_and_notify",
  "status": "awaiting_confirmation"
}
```

`user_id`、`voiceprint_id`、`face_template_id` 仅保存内部 ID，不通过 APP 事件或日志传输原始音频、人脸特征向量和完整模板。

## 6. 事件联动策略

### 6.1 火焰/烟雾：最高优先级

| 条件 | 等级 | 动作 |
|---|---|---|
| 单帧低置信度，或未达到连续帧要求 | `LOW` | 保留截图与事件记录；继续巡检并在后续帧复核 |
| 连续多帧命中，或高风险区域内达到中高置信度 | `MEDIUM` | 停车、抓拍、TTS 提示、APP 待确认推送 |
| 高风险区域内连续高置信度命中，或人工确认升级 | `HIGH` | 急停、录像、声光告警、取消普通 TTS、告警上报、进入 `SAFETY_STOP` |

建议用“连续帧命中 + 冷却时间”减少单帧误报：例如连续 3 帧置信度高于视觉阈值才升级；位于 `high_risk_zone` 时可采用更严格的高等级阈值；同一地点在 30 秒内只创建一个主告警事件，但可持续追加证据。

### 6.2 红绿灯：通行控制

仅在地图中定义 `traffic_control_zone` 的区域使用红绿灯结果，避免把设备指示灯误当作通行灯。

| 结果 | 动作 |
|---|---|
| 红灯 | 发布导航暂停/速度为零，播报一次“前方红灯，等待通行” |
| 黄灯 | 降速至安全速度，禁止进入交叉区域；超时后请求人工确认 |
| 绿灯 | 恢复导航并记录通行事件 |
| 无法识别/结果抖动 | 保守停车，使用最近连续稳定结果；超过超时进入 `EVENT_HOLD` |

### 6.3 车牌：巡检记录和异常核验

车牌识别只在配置的 `plate_checkpoint` 触发。识别结果与车牌白名单/黑名单比对：白名单记录“已核验”，未知车牌创建 `MEDIUM` 待确认事件，黑名单或禁止时段入场创建 `HIGH` 事件并通知 APP。车牌事件本身不应强制急停，除非其所在点位同时被配置为受控出入口。

### 6.4 人脸：双因子认证与敏感操作复核

人脸识别的默认用途是认证，不应持续采集途经人员。以下情况要求重新人脸确认：

- 声纹认证后的初次控制会话建立；
- 从 APP 或语音请求进入“手动接管”；
- 会话超过设定时长后执行“恢复巡检”“应急配送”等敏感动作；
- 管理员执行阈值、白名单或用户录入等配置操作。

## 7. TTS、ASR 与安全优先级

### 7.1 TTS 队列优先级

| 优先级 | 示例 | 队列策略 |
|---|---|---|
| P0 安全 | “警报：发现疑似火情”“已急停” | 立即取消 P1/P2，插队播报，可重复提醒 |
| P1 任务 | “前方红灯，等待通行”“已到达配电柜 A” | 覆盖同类旧播报，不覆盖 P0 |
| P2 交互 | “你好，张三”“请面向摄像头” | 可排队，过期后丢弃 |

### 7.2 ASR 执行规则

- FunASR 2-pass 的首结果仅用于界面/灯效反馈；二次稳定结果才进入意图解析和执行。
- Whisper 仅作为手动切换的备份引擎；切换状态要上报 APP，避免同一语句被两个引擎重复执行。
- 所有危险命令采用确认语义，例如“开始 A 路线巡检”后播报“确认开始 A 路线巡检吗”；用户回答“确认”或在 APP 确认后执行。
- `停止/急停` 使用关键词直通安全通道，不等待二次识别、声纹、人脸或网络返回。

## 8. 实施顺序

### Phase 1：最小可演示闭环

1. 统一声纹结果、人脸结果和视觉结果的 ROS2 消息。
2. 实现 `inspection_multimodal_orchestrator` 的认证状态机：唤醒 -> 声纹 -> 人脸 -> 欢迎语 -> 语音命令放行。
3. 将火焰/烟雾事件接到 `SAFETY_STOP`、TTS P0、抓拍和 APP 上报。
4. 在指定区域接入红绿灯的“停/走”控制，在指定检查点接入车牌记录。
5. 用固定路线完成第 3.3 节的八步演示。

### Phase 2：策略完善与实机标定

1. 完成 ALSA 设备探测与 scipy 重采样，记录每次启动的音频设备和采样率。
2. 在真实小车、不同距离、环境噪声下采集声纹分数，确定接受阈值、拒绝阈值和灰区复核策略。
3. 配置地图区域：`high_risk_zone`、`traffic_control_zone`、`plate_checkpoint`、`face_auth_zone`。
4. 采集正常、疑似和明确火情的视频样本，完成火焰/烟雾视觉阈值、连续帧数与高风险区域策略标定。
5. 向巡检报告加入认证操作记录、红绿灯通行记录、车牌核验结果和告警处置时长。

## 9. 验收用例

| 编号 | 场景 | 预期结果 |
|---|---|---|
| AT-01 | 非授权人员说唤醒词 | 声纹失败；不开放控制权限；不播报身份信息 |
| AT-02 | 授权声纹但错误人脸 | 人脸失败；会话不建立；小车保持静止 |
| AT-03 | 双因子认证成功 | 播报“你好，{姓名}”；生成带角色与超时的会话 |
| AT-04 | 有效会话下开始巡检 | 任务进入运行状态，并向 APP 推送任务 ID |
| AT-05 | 红灯/绿灯切换 | 红灯停车，绿灯恢复；全程写入通行事件 |
| AT-06 | 白名单/未知车牌 | 白名单归档；未知车牌创建待确认事件与截图 |
| AT-07 | 烟雾火焰高危条件 | 小车急停、普通 TTS 被取消、抓拍录像、声光告警、APP 收到高危事件 |
| AT-08 | 任意人员说“急停”或按物理急停 | 不依赖认证，底盘立即停止并记录来源 |
| AT-09 | 会话过期后请求恢复巡检 | 拒绝恢复并要求重新声纹+人脸认证 |

## 10. 风险与边界

- 声纹阈值尚未实机标定前，不应声称具有生产级身份安全性；答辩演示可采用受控环境中的临时阈值。
- 声纹和人脸都是演示场景下的双因子门禁，不代替工业现场的实体门禁、网络权限和安全责任体系。
- 火灾告警始终优先于业务权限；语音认证通过不代表可以忽略火情处置。
- 车牌和人脸涉及个人信息，应仅存必要的 ID、结果、时间和加密/受控证据引用；演示结束后及时清理原始音视频样本。
- 红绿灯识别仅用于预配置的模拟或受控通行区域。开放道路场景必须由具备法规要求的车辆控制系统处理，不能以本模块替代。
