# 车端可部署镜像调研结论

> 目标：把 CI 打出的镜像做成 **小车端可直接 `docker pull` 并运行** 的形态。
> 本文梳理调研结论与前置条件，不含最终实现。

## 一、背景问题

现有 CI 打包流程（`.github/workflows/car-ci.yml` + `docker/Dockerfile.car`）用的是
`python:3.8-slim`，这是 **x86_64（amd64）** 镜像；而小车是 **NVIDIA Jetson（ARM64/aarch64）**。
架构不匹配，导致构建出的镜像**小车端无法直接拉取运行**。

当前这套 CI 的真实作用只是 **源码语法校验 + 可复现的构建产物**（见 Dockerfile 内注释），
并不是一个能在车上跑的镜像。

## 二、运行目标的选定：`api_server.py`

仓库里有两个控车入口，用途完全不同：

| | `app.py`（原厂 Yahboom 遗留） | `api_server.py`（FireGuard 自研，作者：曹晋豪）|
|---|---|---|
| 端口 | 6500 | 5000 |
| 依赖 | `import rosmaster_main`（Cython `.so`，**仓库无源码**）| `from Rosmaster_Lib import Rosmaster`（直接串口）|
| 协议 | 二进制 TCP 6000（`$...#` 帧）| HTTP + JSON REST |
| 可扩展性 | 闭源 `.so`，无法扩展 | 纯 Python，可加传感器/地图/告警 |
| 结论 | 只做基础控车演示 | **项目实际入口，选为镜像运行目标** |

### 为什么选 `api_server.py`

- 纯 Python、无 `.so` 依赖，理论上能干净打进 arm64 镜像。
- 它是 FireGuard 的"业务控制层"，把控车、传感器、SLAM 导航、火情联动串成一条链
  （对应 RAG 文档"车端服务层：控车接口、视频流、状态推送、任务编排"）。
- git 提交脉络印证了这一意图：
  - `712ad6f` 添加 iCar REST API 服务器：状态查询/移动控制/摄像头流
  - `046f02a` 添加 FireGuard 传感器驱动 + REST API + 启动脚本
  - `4109eaf` 添加地图导航功能：选点导航页面 + 地图 API
  - `28f9a0e` 修复：控制面板布局、旋转指令、速度公式、pointerdown 事件

## 三、关键结论：车端部署不需要启动 `app.py`

`api_server.py` **自己就能控车**，不依赖 `app.py`：

```python
from Rosmaster_Lib import Rosmaster   # 直接拿底层串口库
bot = Rosmaster(debug=False)
bot.create_receive_threading()
bot.set_car_motion(vx, vy, vz)        # 直接串口发运动指令
```

它直接调 `Rosmaster_Lib` 走串口（`/dev/myserial`）控底盘，**完全绕过** `app.py`
及其 `rosmaster_main.so`。两者是平行的两套控车实现。

### ⚠️ 两者不能同时运行

`app.py`（内部 `rosmaster_main`）和 `api_server.py` 都通过 `Rosmaster_Lib` 抢同一个
串口 `/dev/myserial`。**同时启动会串口争用**，指令与回包互相打架。
用 `api_server.py` 就**不要**再起 `app.py`，二选一。

## 四、能力边界：哪些镜像自足，哪些依赖车上 ROS2 容器

| 功能 | 接口 | 实现方式 | 镜像自身是否够 |
|---|---|---|---|
| **控车** | `/api/move` | `Rosmaster_Lib` 直接串口 | ✅ 自足 |
| **传感器** | `/api/status` `/api/sensors` | `iCarSensorDriver` 读 `/dev/ttyUSB2` | ✅ 自足 |
| **视频流** | `/video_feed` | `subprocess` curl 车上 `localhost:8080`（ROS2 `web_video_server`）| ❌ 依赖 ROS2 容器 |
| **地图/导航** | `/api/map_image` `/api/navigate` | `docker exec` 进 `icar/ros-foxy` 发 Nav2 | ❌ 依赖 ROS2 容器 |

即：`api_server.py` 构建的包**自己能完成控车 + 传感器**；
视频与 SLAM 导航是它用 `docker exec` / `curl` 去调**车上另一个 ROS2 容器**
（`start_services.sh` 启动的 `icar/ros-foxy` / `fireguard_cam`），这部分不在镜像内。

### 车端部署完整图景

```
车（Jetson 宿主机）
├── [你的镜像] api_server.py 容器  ← 控车 + 传感器 + 转发
│      需要: --network host, -v /dev:/dev, 挂 docker.sock
│      里面必须打进: Rosmaster_Lib（仓库里没有，得 COPY 进去）
│
└── [车上已有] ROS2 容器 icar/ros-foxy  ← 视频流 :8080 + Nav2 导航
       由 start_services.sh 启动，不是你打的镜像
```

## 五、做成"车端可直接拉取运行"镜像的前置条件

1. **按 `linux/arm64` 构建** — GitHub 默认 runner 是 x86。二选一：
   - QEMU + buildx 交叉构建（`docker/setup-qemu-action` + `platforms: linux/arm64`）；
   - 或使用 GitHub 原生 arm64 runner（`ubuntu-22.04-arm`）。
2. **arm64 基础镜像**：
   - 控制 + 视频转发这类只需 `arm64/python:3.8-slim` 即可；
   - 若要 GPU/CUDA（火焰检测 torch），需与车上 **JetPack/L4T 版本匹配**的 L4T 基础镜像
     （车上执行 `cat /etc/nv_tegra_release` 或 `dpkg -l | grep nvidia-l4t` 获取版本）。
3. **把硬件库打进镜像**：
   - `Rosmaster_Lib`（纯 Python 串口库，仓库未 track，需 `COPY` 进镜像或车端 `-v` 挂载）；
   - `sensors/icar_sensor_driver.py` 传感器驱动（仓库已有）。
4. **运行期设备访问**：容器需 `--network host`、`-v /dev:/dev`（或 `--device /dev/myserial`
   等具体设备），大概率还要 `--privileged`；若要转发/操作 ROS2 容器还需挂 `docker.sock`。
5. **车能访问的镜像仓库**：推送到 `ghcr.io` 作为可 `docker pull` 的包（公开，或车端
   `docker login` 用 token 拉私有包）。

## 六、待确认事项（进入实现前需要拍板）

- **arm64 构建方式**：QEMU 交叉构建 vs 原生 arm64 runner。
- **基础镜像层**：普通 arm64 python（仅控车/传感器）vs L4T（需 GPU）。
- **`Rosmaster_Lib` 进镜像方式**：`COPY` 打包 vs 车端 `-v` 挂载。
- **镜像范围**：只做"纯控车 + 传感器"最小自足镜像，还是连视频/导航一起
  （后者要求车上 ROS2 容器先跑起来）。
- **车上 JetPack / L4T 版本号**（仅当需要 GPU 时必需）。
