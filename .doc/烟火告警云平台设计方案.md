# 烟火告警云平台设计方案（Plan）

> 状态：设计阶段（Plan）。本文档只做方案设计，不含实现代码。
> 目标：把车端烟火检测产生的**确认告警事件**上报到云端 MySQL，为后续前后端展示提供数据源。

---

## 一、背景与范围

### 1.1 已确认的四项决策

| 决策项 | 选定方案 |
|--------|----------|
| 覆盖模块 | **先只做烟火**（红绿灯只是 CV 状态、车牌无真实识别，暂不纳入） |
| 上报链路 | **车端 → 云端接收 API → MySQL**（车端不直连数据库） |
| 证据图片 | **图片上传对象存储，DB 只存 URL** |
| 上报粒度 | **只上报确认的告警事件**（不上报逐帧检测） |

### 1.2 为什么只做烟火（代码现实）

经代码核查，三个"检测"成熟度差异极大：

- **烟火**：真·YOLOv5 模型 + 完整事件状态机 + AI 二次复核，产出带唯一 ID、时间戳、置信度、证据图片的结构化事件。**唯一天然适合入库的模块。**
- **红绿灯**：纯 OpenCV（HSV + 霍夫圆），只返回 `red/yellow/green/none` 字符串，代码里无 ID、无时间戳、无置信度。
- **车牌**：`demo_showcase` 里仅用 Haar 级联画框标 "CHINA PLATE"，**不产出任何车牌号**，无数据源。

因此本期只落地烟火，表结构预留扩展空间（见 §4.6）。

**关键设计原则**：车端只多一个可插拔的 `AlarmService` 实现，**不侵入检测主流程**。现有 `LoggingAlarmService` 保留作为本地兜底，云端上报作为额外通道叠加。

## 三、车端集成设计

### 3.1 集成点（已代码核实）

告警在 `event_manager.py:119-129` 通过以下三个方法发射，参数均为完整的 `AlarmEvent`：

- `report_confirmed_fire(event)` → 确认明火
- `report_suspected_smoke(event)` → 疑似烟雾
- `report_ai_unavailable(event)` → AI 失效需人工介入

注入点在 `detect.py:56`：
```python
LoggingAlarmService(monitor_config.runtime_dir, monitor_logger)
```

### 3.2 `AlarmEvent` 可用字段（数据源，types.py:50-58）

| 字段 | 类型 | 说明 |
|------|------|------|
| `event_id` | str | 唯一 ID，如 `fire_20260712_143052_123456_001` |
| `alarm_type` | str | `confirmed_fire` / `suspected_smoke` / `ai_unavailable` |
| `occurred_at` | datetime | 带时区，事件发生时刻 |
| `reason` | str | AI 复核给出的中文原因 |
| `confidence` | float? | AI 置信度 0~1（`ai_unavailable` 时为 None） |
| `evidence_path` | str? | 本地证据图片路径（用于上传对象存储） |
| `local_detection_gone` | bool | 告警时本地目标是否已消失 |

补充可从 `event_manager._metadata()` 拿到的检测侧信息：
`local_detection.classes`、`local_detection.max_confidence`、`trigger_hits`、`window_seconds`。

### 3.3 新增 `CloudAlarmService`（设计，不含实现）

- **接口**：实现现有 `AlarmService` Protocol（三个 `report_*` 方法），可与 `LoggingAlarmService` 组成"复合上报"（先本地、后云端）。
- **职责**：
  1. 读取 `evidence_path` 图片 → 上传对象存储 → 得到公网可访问 URL。
  2. 组装告警 JSON（§5.1）→ POST 到云端接收 API。
  3. **断网容错**：上报失败时写入本地 `runtime/cloud_outbox.jsonl`，后台线程按退避策略重试；成功后移除。
  4. **幂等**：以 `event_id + alarm_type` 为业务主键，重复上报由云端去重（§4.2 唯一索引）。
- **配置**（沿用 `.env` 风格，参考 config.py）：
  - `CLOUD_API_BASE_URL`、`CLOUD_API_TOKEN`（鉴权）
  - `CLOUD_UPLOAD_TIMEOUT_SECONDS`、`CLOUD_RETRY_MAX`
  - 对象存储相关：`OBJECT_STORE_*`
- **不阻塞检测**：上报走独立线程/队列，绝不拖慢 YOLO 主循环（复用现有 AIReviewer 的异步模式思路）。

## 四、数据库设计（MySQL `dev_fireguard`）

> 连接信息：`8.140.28.233:3306`，库 `dev_fireguard`，用户 `root`。
> ⚠️ 安全提醒：生产环境**不应用 root + 弱口令直连**，且接收 API 才连库、车端不直连（见 §7）。

### 4.1 主表 `fire_alarm`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 代理主键 |
| `event_id` | VARCHAR(64) | NOT NULL | 车端事件 ID |
| `alarm_type` | VARCHAR(32) | NOT NULL | confirmed_fire/suspected_smoke/ai_unavailable |
| `occurred_at` | DATETIME(6) | NOT NULL | 事件发生时刻（车端时间） |
| `reason` | VARCHAR(300) | NULL | AI 复核原因 |
| `confidence` | DECIMAL(4,3) | NULL | 0.000~1.000 |
| `evidence_url` | VARCHAR(512) | NULL | 对象存储图片 URL |
| `detection_classes` | VARCHAR(64) | NULL | 命中类别，如 `fire,smoke` |
| `max_confidence` | DECIMAL(4,3) | NULL | 本地检测最大置信度 |
| `local_detection_gone` | TINYINT(1) | NOT NULL DEFAULT 0 | 告警时本地目标是否消失 |
| `car_id` | VARCHAR(32) | NULL | 车辆标识（多车预留，本期可固定值） |
| `received_at` | DATETIME(6) | NOT NULL | 云端入库时刻（服务器时间） |
| `raw_payload` | JSON | NULL | 原始上报 JSON 存档，便于追溯/加字段 |

### 4.2 索引

- `UNIQUE KEY uk_event_type (event_id, alarm_type)` —— **幂等去重的核心**，重复上报直接 `INSERT ... ON DUPLICATE KEY UPDATE` 或忽略。
- `KEY idx_occurred (occurred_at)` —— 前端按时间倒序查询。
- `KEY idx_type (alarm_type)` —— 按告警类型筛选。

### 4.3 字符集

`utf8mb4` / `utf8mb4_unicode_ci`（reason 含中文）。

### 4.4 建表 SQL（草案）

```sql
CREATE TABLE IF NOT EXISTS fire_alarm (
  id                    BIGINT       NOT NULL AUTO_INCREMENT,
  event_id              VARCHAR(64)  NOT NULL,
  alarm_type            VARCHAR(32)  NOT NULL,
  occurred_at           DATETIME(6)  NOT NULL,
  reason                VARCHAR(300) NULL,
  confidence            DECIMAL(4,3) NULL,
  evidence_url          VARCHAR(512) NULL,
  detection_classes     VARCHAR(64)  NULL,
  max_confidence        DECIMAL(4,3) NULL,
  local_detection_gone  TINYINT(1)   NOT NULL DEFAULT 0,
  car_id                VARCHAR(32)  NULL,
  received_at           DATETIME(6)  NOT NULL,
  raw_payload           JSON         NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_event_type (event_id, alarm_type),
  KEY idx_occurred (occurred_at),
  KEY idx_type (alarm_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 4.5 为什么保留 `raw_payload`（JSON）

上报字段未来可能增补（如 GPS、车速）。整包存档可在不改表的情况下先落地，稳定后再抽列，降低迭代成本。

### 4.6 扩展预留（本期不建）

若后续纳入红绿灯/车牌，建议**每模块独立表**（结构差异大），而非塞进 `fire_alarm`。可共用 `car_id`、`occurred_at`、`evidence_url` 等通用列的命名约定。

## 五、接口契约

### 5.1 上报接口 `POST /api/v1/fire-alarms`

请求体（车端 → 云端）：
```json
{
  "event_id": "fire_20260712_143052_123456_001",
  "alarm_type": "confirmed_fire",
  "occurred_at": "2026-07-12T14:30:52.123456+08:00",
  "reason": "画面中部可见明显跳动火焰",
  "confidence": 0.91,
  "evidence_url": "https://<store>/fire_xxx_001_initial_143052.jpg",
  "detection_classes": ["fire"],
  "max_confidence": 0.88,
  "local_detection_gone": false,
  "car_id": "car-01"
}
```

响应：
```json
{ "code": 0, "msg": "ok", "id": 123, "duplicated": false }
```

- 鉴权：请求头 `Authorization: Bearer <CLOUD_API_TOKEN>`。
- 幂等：命中 `uk_event_type` 时返回 `duplicated: true`，不报错。
- 图片先传对象存储再调本接口，body 只带 URL（决策：DB 存 URL）。

### 5.2 图片上传

- 本期推荐 **方案A**：云主机建目录（如 `/data/fireguard/evidence/`），Nginx 反代静态目录对外提供 URL。接收 API 增加 `POST /api/v1/evidence`（multipart）接收图片、返回 URL。
- 可扩展 **方案B**：MinIO / 阿里云 OSS，`CloudAlarmService` 直传，拿回 URL。

### 5.3 查询接口（供前端，后续实现）

- `GET /api/v1/fire-alarms?type=&from=&to=&page=&size=` 分页 + 筛选。
- `GET /api/v1/fire-alarms/{id}` 单条详情。

## 六、云端服务技术选型

| 项 | 选型 | 理由 |
|----|------|------|
| Web 框架 | Flask（与仓库现有 `api_server.py` 一致） | 团队已用，学习成本低 |
| DB 驱动 | PyMySQL / SQLAlchemy | 纯 Python，易部署 |
| 部署 | 云主机直跑 或 Docker | 与现有 CI 打包思路衔接 |
| 图片服务 | Nginx 静态目录（方案A） | 最快落地 |

## 七、安全与容错要点

1. **车端不直连 MySQL**：所有写库经接收 API，避免把库凭据下发到车端、避免公网暴露 3306。
2. **数据库账号**：为接收 API 单独建最小权限账号（仅 `fire_alarm` 表 `INSERT/SELECT/UPDATE`），勿用 root。
3. **接口鉴权**：Bearer Token，防止告警被伪造。
4. **断网容错**：车端 `cloud_outbox.jsonl` 本地队列 + 后台重试，恢复网络后补传，保证不丢告警。
5. **幂等**：`event_id + alarm_type` 唯一索引，重试/重复上报不产生重复记录。
6. **不阻塞检测**：上报异步化，YOLO 主循环零等待。
7. **时区**：车端上报 ISO8601 带时区；云端另记 `received_at`（服务器时间）便于排查时钟漂移。

## 八、落地步骤（Roadmap）

1. **云端 DB**：连库、建 `fire_alarm` 表（§4.4）、建最小权限账号。
2. **云端接收 API**：实现 `POST /fire-alarms` + `POST /evidence` + 幂等写库；Nginx 静态图片目录。
3. **车端 `CloudAlarmService`**：实现 `AlarmService` 接口，图片上传 + 上报 + 本地重试队列；在 `detect.py:56` 与 `LoggingAlarmService` 组合注入。
