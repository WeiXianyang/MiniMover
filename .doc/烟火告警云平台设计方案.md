# 烟火告警云平台设计方案（已实现）

> 状态：已实现。车端上报 + 云端接收 API 均已完成编码。
> 目标：把车端烟火检测产生的**确认告警事件**上报到云端 MySQL，为后续前后端展示提供数据源。

---

## 一、背景与范围

### 1.1 决策汇总

| 决策项 | 选定方案 |
|--------|----------|
| 覆盖模块 | **只做烟火**（红绿灯只是 CV 状态、车牌无真实识别，暂不纳入） |
| 上报链路 | **车端 → 云端接收 API → MySQL**（车端不直连数据库） |
| 证据图片 | **图片上传 Nginx 静态目录，DB 只存 URL** |
| 上报粒度 | **只上报确认的告警事件**（不上报逐帧检测） |
| 车辆标识 | **多车动态区分**（每车 .env 配置 CAR_ID） |
| 技术栈 | **Flask + PyMySQL**（与仓库现有 api_server.py 一致） |

### 1.2 为什么只做烟火

经代码核查，三个"检测"成熟度差异极大：

- **烟火**：真·YOLOv5 模型 + 完整事件状态机 + AI 二次复核，产出带唯一 ID、时间戳、置信度、证据图片的结构化事件。**唯一天然适合入库的模块。**
- **红绿灯**：纯 OpenCV（HSV + 霍夫圆），只返回 `red/yellow/green/none` 字符串，代码里无 ID、无时间戳、无置信度。
- **车牌**：`demo_showcase` 里仅用 Haar 级联画框标 "CHINA PLATE"，**不产出任何车牌号**，无数据源。

## 四、车端 CloudAlarmService 详解

### 4.1 两个类

**`CompositeAlarmService(*services)`**（串联模式）
- 构造函数接受任意多个 `AlarmService` 实例。
- 每个 `report_*` 调用遍历所有子 service，全部调用，无短路。
- 合成返回值：任一子 service 返回 `True` 则结果为 `True`。

**`CloudAlarmService(runtime_dir, logger)`**（云端上报）
- 实现 `AlarmService` Protocol（3 个 `report_*` 方法）。
- 构造时自动启动后台 daemon 线程 `cloud-alarm`。

### 4.2 线程模型（不阻塞检测）

```
report_confirmed_fire/...  调用线程 (detect.py 主循环)
    │
    ├── 去重检查 (event_id, alarm_type) → 已发过则 return False
    ├── 组装 payload → queue.put_nowait()  →  return True
    │                                           ▼
    └── 队列满降级 → 写 outbox.jsonl 兜底 ───→ 后台线程读文件重试
                                                        │
                                              _run() 后台线程
                                                │
                                                ├── queue.get() 阻塞等任务
                                                ├── _process_one(entry)
                                                │     ├── evidence_path 存在 → _upload_evidence()
                                                │     └── _post_alarm()
                                                ├── 失败 → outbox.jsonl
                                                └── _retry_outbox() 指数退避重试
```

### 4.3 配置（车端 `.env` 新增）

```env
CLOUD_API_BASE_URL=http://8.140.28.233:8000   # 云端 API 根地址
CLOUD_API_TOKEN=<your-token>                   # 鉴权 Bearer token
CAR_ID=car-01                                  # 车辆标识（多车动态区分）
CLOUD_RETRY_MAX_SEC=300                        # 最大重试秒数（指数退避上限）
```

### 4.4 断网容错

- **实时上报失败** → 条目保留在 outbox 队列中，不丢弃。
- **后台重试** → 每处理一个实时任务后，顺带扫描 `runtime/cloud_outbox.jsonl`，逐条以指数退避（2s → 4s → 8s → ... → cap at `CLOUD_RETRY_MAX_SEC`）重试。
- **成功清理** → 重试成功后从 outbox 移除；全部成功则删除 outbox 文件。
- **进程关闭** → `close()` 等待队列排空（grace 5s），剩余条目留在 outbox 供下次进程启动时重试。

### 4.5 HTTP 传输

- 优先使用 `requests` 库（更简洁）。
- `requests` 不可用时自动退化为纯 `urllib.request`（Jetson 最小依赖，无需额外 pip 安装）。

## 五、数据库设计（MySQL `dev_fireguard`）

> 连接信息：`8.140.28.233:3306`，库 `dev_fireguard`。
> 建表方式：`mysql -h 8.140.28.233 -u root -p dev_fireguard < cloud_platform/schema.sql`
> ⚠️ 安全提醒：生产环境**不应用 root + 弱口令直连**，且接收 API 才连库、车端不直连。

### 5.1 主表 `fire_alarm`

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 代理主键 |
| `event_id` | VARCHAR(64) | NOT NULL | 车端事件 ID |
| `alarm_type` | VARCHAR(32) | NOT NULL | confirmed_fire / suspected_smoke / ai_unavailable |
| `occurred_at` | DATETIME(6) | NOT NULL | 事件发生时刻（车端时间，带微秒） |
| `reason` | VARCHAR(300) | NULL | AI 复核原因 |
| `confidence` | DECIMAL(4,3) | NULL | AI 置信度 0.000~1.000 |
| `evidence_url` | VARCHAR(512) | NULL | Nginx 静态图片 URL |
| `detection_classes` | VARCHAR(64) | NULL | 命中类别，如 `fire,smoke` |
| `max_confidence` | DECIMAL(4,3) | NULL | 本地检测最大置信度 |
| `local_detection_gone` | TINYINT(1) | NOT NULL DEFAULT 0 | 告警时本地目标是否消失 |
| `car_id` | VARCHAR(32) | NOT NULL DEFAULT 'unknown' | 车辆标识（多车动态区分） |
| `received_at` | DATETIME(6) | NOT NULL | 云端入库时刻（服务器时间） |
| `raw_payload` | JSON | NULL | 原始上报 JSON 存档（便于追溯/加字段） |

### 5.2 索引

- `UNIQUE KEY uk_event_type (event_id, alarm_type)` —— **幂等去重的核心**。
- `KEY idx_occurred (occurred_at)` —— 前端按时间倒序查询。
- `KEY idx_type (alarm_type)` —— 按告警类型筛选。
- `KEY idx_car (car_id)` —— 多车筛选。

### 5.3 为什么保留 `raw_payload`（JSON）

上报字段未来可能增补（如 GPS、车速）。整包存档可在不改表的情况下先落地，稳定后再抽列，降低迭代成本。

## 六、接口契约

### 6.1 `POST /api/v1/evidence` — 图片上传

请求：multipart/form-data
- `file`: JPEG/PNG 图片文件
- `event_id`: 可选，用于命名

响应：
```json
{ "code": 0, "msg": "ok", "url": "http://8.140.28.233/evidence/fire_20260712_143052_123456_001_a1b2c3d4.jpg" }
```

### 6.2 `POST /api/v1/fire-alarms` — 告警入库

请求体（车端 → 云端）：
```json
{
  "event_id": "fire_20260712_143052_123456_001",
  "alarm_type": "confirmed_fire",
  "occurred_at": "2026-07-12T14:30:52.123456+08:00",
  "reason": "画面中部可见明显跳动火焰",
  "confidence": 0.91,
  "evidence_url": "http://8.140.28.233/evidence/fire_xxx.jpg",
  "detection_classes": ["fire"],
  "max_confidence": 0.88,
  "local_detection_gone": false,
  "car_id": "car-01"
}
```

响应：
```json
{ "code": 0, "msg": "ok", "id": 123, "duplicated": false }
```

- 鉴权：请求头 `Authorization: Bearer <API_TOKEN>`。
- 幂等：命中 `uk_event_type` 时返回 `duplicated: true`，不报错、不重复插入。

### 6.3 `GET /api/v1/fire-alarms` — 分页查询

参数：`?type=confirmed_fire&car_id=car-01&from=2026-07-01T00:00:00&to=2026-07-31T23:59:59&page=1&size=20`

响应：
```json
{ "code": 0, "msg": "ok", "data": { "total": 5, "page": 1, "size": 20, "items": [...] } }
```

### 6.4 `GET /api/v1/fire-alarms/hourly-stats` — 24 小时趋势

参数（可选）：`?from=2026-07-01T00:00:00&to=2026-07-14T23:59:59`

响应：
```json
{
  "code": 0, "msg": "ok",
  "data": [
    {"hour": "00:00", "count": 0},
    {"hour": "01:00", "count": 0},
    {"hour": "14:00", "count": 3},
    ...
    {"hour": "23:00", "count": 1}
  ]
}
```

- 返回 24 个元素（覆盖 00:00~23:00），无数据的时段 `count=0`。
- 按 `occurred_at` 的 `HOUR()` 聚合，默认时间范围最近 24 小时。

### 6.5 `GET /api/v1/vehicles/status` — 车辆最近上报

响应：
```json
{
  "code": 0, "msg": "ok",
  "data": [
    {"car_id": "car-01", "last_seen": "2026-07-14T14:30:55.654321"},
    {"car_id": "car-02", "last_seen": "2026-07-14T13:20:44.321000"}
  ]
}
```

- 按 `car_id` 分组取 `MAX(received_at)`。
- 前端根据 `last_seen` 判断在线/离线（2 分钟内视为在线）。

### 6.6 `GET /api/v1/fire-alarms/{id}` — 单条详情

响应含 `raw_payload`（原始上报 JSON）。

### 6.7 `GET /healthz` — 健康检查

```json
{ "code": 0, "msg": "ok", "db": "up" }
```

## 八、安全与容错

1. **车端不直连 MySQL**：所有写库经接收 API，避免把库凭据下发到车端。
2. **接口鉴权**：Bearer Token，写/读接口均校验（`/healthz` 除外）。
3. **断网容错**：车端 `cloud_outbox.jsonl` 本地队列 + 后台指数退避重试。
4. **幂等**：`UNIQUE KEY uk_event_type(event_id, alarm_type)` 防止重复记录。
5. **不阻塞检测**：上报走独立 daemon 线程，YOLO 主循环零等待。
6. **时区**：车端上报 ISO8601 带时区；云端另记 `received_at`（服务器时间）便于排查时钟漂移。
7. **最小依赖**：CloudAlarmService 同时支持 `requests` 和纯 `urllib`（Jetson 无需额外 pip）。
