# 内科与外科地图标注设计

**日期：** 2026-07-17
**状态：** 已获用户确认，待实现

## 目标

在车端“地图巡逻控制台”中提供两个可持久化的展示标注：`内科` 与 `外科`。操作员选择科室后点击地图，即可保存该科室的地图坐标，并在地图和侧栏显示标签与坐标。

本功能用于比赛演示中的科室可视化，不负责自动导航。

## 非目标与安全边界

- 不创建、发送或取消 Nav2 导航目标。
- 不发布速度、底盘、巡逻路线、初始位姿或任何 ROS 控制消息。
- 不修改既有“开始巡逻”“停止巡逻”“设起点”“画路径”行为。
- 不将标注坐标自动转换为导航终点。

## 推荐方案

采用服务端 JSON 持久化而非浏览器本地存储。标注在 Jetson 的导航项目内独立保存，浏览器刷新、重新打开或切换客户端后仍可见。

## 数据模型

配置文件：`navigation/data/department_markers.json`

```json
{
  "version": 1,
  "markers": {
    "internal_medicine": {"label": "内科", "x": 0.0, "y": 0.0},
    "surgery": {"label": "外科", "x": 0.0, "y": 0.0}
  }
}
```

- 仅允许两个固定键：`internal_medicine` 与 `surgery`。
- 初始文件可为空标注对象；未设定的科室不在地图上显示。
- 写入同一科室会原子覆盖旧坐标。
- 坐标为地图坐标系中的有限数值 `x`、`y`；此功能不存储或使用 `yaw`。

## HTTP 接口

### 读取标注

`GET /api/nav/department-markers`

返回已保存的标注。不存在配置文件时返回成功的空标注集合。

### 保存标注

`POST /api/nav/department-markers`

请求体：

```json
{"department": "internal_medicine", "x": 1.25, "y": -0.5}
```

服务端验证：

- `department` 必须为 `internal_medicine` 或 `surgery`。
- `x`、`y` 必须可转换为有限浮点数。
- 验证失败时返回清晰错误，不写入文件。

接口只读写标注 JSON，不调用 ROS、Docker、Nav2、底盘或巡逻模块。

## 页面交互

在现有 `导航栈` 控件旁新增三个纯展示控件：

1. `标注内科`
2. `标注外科`
3. `取消标注`

流程：

1. 点击一个科室标注按钮后，页面进入该科室的单次点选模式。
2. 操作员在地图有效区域点击，复用现有画布像素到地图坐标的转换逻辑。
3. 页面调用标注保存接口；成功后立即绘制名称标签与颜色标记，并刷新侧栏坐标。
4. 若接口失败，保留原有显示状态并给出错误提示。
5. 新标注同一科室会覆盖旧位置；无需额外复核弹窗，以满足比赛现场的快速操作需求。
6. 页面初始化时读取标注接口并渲染已保存的标记。

建议颜色：内科为蓝色、外科为橙色，以便快速区分。具体视觉样式不影响导航行为。

## 组件边界

- `navigation/department_markers.py`：配置加载、校验、原子保存的无 ROS 依赖逻辑。
- `navigation/routes.py`：HTTP 路由，仅编排请求校验与响应。
- `navigation/patrol_page.py`：显示控件、地图覆盖层和前端请求逻辑。
- `navigation/data/department_markers.json`：运行时持久化数据；提供初始空模板或由加载逻辑自动创建。

## 错误处理

- 配置文件缺失：视作无标注，不阻塞页面。
- 配置文件格式损坏：接口返回错误并保留已有文件，不自动覆盖。
- 非法科室或坐标：HTTP 400，不写入。
- 文件写入失败：HTTP 500，页面保留旧标注并提示保存失败。

## 验证策略

1. 单元测试：验证白名单、有限数值校验、缺失文件读取、覆盖保存和损坏 JSON 的处理。
2. 路由测试：验证 GET 与 POST 成功响应及非法输入响应。
3. 页面静态检查：确认标注模块未调用导航、巡逻、速度或底盘控制接口。
4. 手动浏览器验证：分别保存内科、外科；刷新页面后确认两个标签和坐标仍存在。
