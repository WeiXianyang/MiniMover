# 导航 / 定点配送 对接说明

> 本文单独说明「导航（定点配送 + 单点导航）」模块在 App 侧的对接实现：
> 新增了哪些页面、改了哪些页面的跳转、对接了哪些车端接口、以及数据流。
> 车端接口以《小车端功能对接文档》2.4 节 `/api/nav/*` 为准。

---

## 一、总览

导航能力在 App 里拆成两条路径，服务两种场景：

| 场景 | 页面 | 交互 | 适合 |
|------|------|------|------|
| **多途经点巡检** | 配送页（Delivery，标签页） | 选已保存路线 / 自选固定点 → 开始巡逻 | 标准巡检、批量配送 |
| **单点快速导航** | 地图导航页（MapNav，路由页） | 地图上点一个点 → 下发目标 | 临时去某处 |

设计原则：**手机端不做"在栅格地图上逐点点选"**（小屏精确定位体验差），多点路线走"选方案 / 选固定点"，单点导航才用地图点选。

---

## 二、新增页面

### 1. `pages/ScreenMapNav`（地图导航页，新增）

- **类型**：`@Entry` 路由页（非标签页），竖屏
- **用途**：加载车端真实栅格地图，点选一个目标点下发单点导航
- **注册**：已加入 `main_pages.json` 的 `src` 列表
- **对接接口**：
  - `GET /api/nav/map`（或兼容旧版 `/api/map`）— 地图元数据（宽高/分辨率/原点）
  - `GET /api/nav/map/image`（或 `/api/map_image`）— 地图 JPEG，直接给 `Image` 组件
  - `POST /api/nav/coord/pixel_to_map` — 像素→世界坐标换算（也可本地按元数据算）
  - `POST /api/nav/initial_pose` / `POST /api/navigate` — 下发目标点
- **关键逻辑**：点选地图 → 记录像素坐标 → 按 `origin + px*resolution` 换算世界坐标 → 下发
- **前置条件**：仅 REST 模式可用；未连接时显示"未连接 REST 设备，地图不可用"

> 注：配送页（Delivery）本身是已有标签页，不算新增，只是重写了内容（见第四节）。

## 四、配送页（Delivery）重写

原为静态 mock，现对接 `/api/nav/patrol/*`，双模式：

### 模式 A：选择已保存路线
- `GET /api/nav/patrol/routes` — 拉车端已保存路线名列表（如 `campus`/`warehouse`）
- 卡片列表展示，点击某条 → `GET /api/nav/patrol/routes/<name>` 取途经点 → `POST /api/nav/patrol/route` 设为当前路线

### 模式 B：自选预设固定点
- 页面内置若干固定巡检点（配电柜 A/B、仓储通道、消防器材点、充电待命点、应急物资点 D），2 列网格展示
- 点击加入"自选路线"面板（可删、可清空、显序号）
- "设为此路线" → `POST /api/nav/patrol/route` 下发途经点数组
- **预设点坐标可按实际地图替换，不改 UI 逻辑**

### 通用控制（轮询 + 按钮）
- 每 4 秒轮询 `GET /api/nav/patrol/status` + `GET /api/nav/stack/status`
- 导航栈未就绪 → "启动导航栈"按钮 → `POST /api/nav/stack/start`
- 已就绪 + 已设路线 → "开始巡逻"`POST /api/nav/patrol/start` / "停止巡逻"`POST /api/nav/patrol/stop`
- 巡逻中 → 进度条（`navigating_waypoint_N` → 百分比）

## 五、对接的车端接口清单

全部在 REST :5000 的 `/api/nav/` 命名空间下，统一 `{code,msg,data}` 包络（`code===0` 成功）。

| # | 方法 | 路径 | 用途 | App 调用位置 |
|---|------|------|------|-------------|
| 1 | GET | `/api/nav/map` | 地图元数据 | 地图页 |
| 2 | GET | `/api/nav/map/image` | 地图图片 JPEG | 地图页 |
| 3 | POST | `/api/nav/coord/pixel_to_map` | 像素→世界坐标 | 地图页 |
| 4 | POST | `/api/nav/initial_pose` | 设 AMCL 初始位姿 | 地图页 |
| 5 | GET | `/api/nav/stack/status` | 导航栈状态 | 配送页轮询 |
| 6 | POST | `/api/nav/stack/start` | 启动导航栈 | 配送页 |
| 7 | POST | `/api/nav/stack/stop` | 关闭导航栈 | （预留） |
| 8 | GET | `/api/nav/patrol/route` | 当前巡逻路线 | 配送页轮询 |
| 9 | POST | `/api/nav/patrol/route` | 设置巡逻路线 | 配送页两模式 |
| 10 | GET | `/api/nav/patrol/status` | 巡逻运行状态 | 配送页轮询 |

## 七、数据流

### 巡检路线（配送页）
```
配送页进入
  → 轮询 stack/status + patrol/status + patrol/route + patrol/routes
  → 用户选路线（已保存 or 自选固定点）
  → POST patrol/route 设为当前路线
  → 若栈未就绪：POST stack/start（约 15s）
  → POST patrol/start 发车
  → 轮询 patrol/status 刷新进度条
```

### 单点导航（地图页）
```
主页/配送页/接管页 → pushUrl 地图页
  → GET nav/map + nav/map/image 加载地图
  → 用户点选地图
  → 像素坐标 → 世界坐标换算
  → POST navigate / initial_pose 下发
  → router.back() 返回
```

---

## 八、注意事项

1. **仅 REST 模式**：导航全套接口都在 :5000，TCP(6000) 模式下地图/巡逻不可用。
2. **导航栈需先启动**：`stack_ready=false` 时先 `stack/start`，异步约 15s，期间轮询状态。
3. **预设点坐标要按实际地图改**：配送页模式 B 的固定点是示例值，需替换成真实 SLAM 地图坐标（可在车端 `/nav` 网页点图取坐标）。
4. **路线创建在车端**：模式 A 的已保存路线由车端 `/nav/patrol` 网页或后台配置，App 只做加载执行。
5. **无实时位姿接口**：巡逻进度用 `navigating_waypoint_N` 估算百分比，非精确里程。
