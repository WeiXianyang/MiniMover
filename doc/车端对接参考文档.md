# FireGuard 巡检 APP — 车端对接参考文档

> 本文总结本 OpenHarmony App 与小车端（Jetson）+ 云平台的全部对接方式：功能、页面、
> API、端口、数据格式与关键决策。供**另一个页面相同、架构不同的安卓端 App** 对接时参考。
> 不涉及本项目具体代码位置，只讲"怎么接"。

---

## 一、通信端口总览

小车/云平台对外暴露的通道，App 实际用到的已标注：

| 端口 | 协议 | 用途 | App 是否使用 |
|------|------|------|:---:|
| **5000** | HTTP REST (Flask) | 状态查询、底盘控制、视频代理、地图导航、音频 | ✅ 主用 |
| **5001** | TCP 纯文本 | 低延迟底盘控制（旁路 HTTP，~2-5ms） | ✅ 摇杆/急停 |
| **6000** | TCP 自定义 Hex 帧 | 旧版遥控协议，能力最全（灯带/机械臂/四轮/拍照） | ✅ 兼容模式 |
| **8080** | HTTP MJPEG | ROS2 `web_video_server` 相机原流 | ✅ 视频 |
| **6500** | HTTP MJPEG | 旧版 Web 视频服务 | ❌ 已弃用 |
| **8000** | HTTP REST | 云平台火情告警（独立服务器 `8.140.28.233`） | ✅ 告警页 |

> **注意**：5000/5001/8080 都在**小车本机 IP** 上；8000 是**云平台独立地址**，与小车不同机。

## 二、两种后端连接模式

App 支持两种底盘控制后端，用户在"连接"页切换：

### 模式 A：TCP（旧版，端口 6000）
- 自定义 ASCII-Hex 帧协议：`$ + 车型(2) + 命令码(2) + 长度(2) + 数据 + 校验(2) + #`
- 长连接、单向（只发不收）、无心跳、惰性重连
- 能力最全：灯带、机械臂、四轮独立、拍照录像都在这条通道
- 适合完全兼容旧车

### 模式 B：REST（现代，端口 5000 + 5001 + 8080）
- HTTP/JSON 做状态查询、地图、音频等
- 底盘高频控制走 5001 低延迟 TCP 旁路（连不上自动回退 5000 HTTP）
- 视频走 8080 ROS 原流
- 推荐新对接方案

App 内用一个全局状态对象记录：当前模式、连接状态、小车 IP、各端口、5001 是否连通。
另一端可用等价的全局单例/状态管理承载这些字段。

## 三、REST API（端口 5000）统一约定

### 响应包络
所有 5000 接口统一返回：

```json
{ "code": 0, "msg": "ok", "data": { ... } }
```

- `code === 0` 为成功，其他为失败
- 解析时先判 `code`，再取 `data`；`data` 可能缺省，按空对象处理
- 建议给每个请求设 5 秒连接/读取超时

### 用到的接口清单

| 方法 | 路径 | 用途 | 请求体 / 参数 |
|------|------|------|------|
| GET | `/api/health` | 健康检查（连接测试） | 无 |
| GET | `/api/status` | 全状态：传感器 + 电量 + IP | 无 |
| GET | `/api/sensors` | 仅传感器数据 | 无 |
| POST | `/api/move` | 底盘移动 | `{cmd, speed, duration}` |
| GET | `/api/map` | 地图元数据 | 无 |
| GET | `/api/map_image` | 地图图片（image/jpeg） | 无 |
| POST | `/api/navigate` | 下发导航目标 | `{x, y, theta}` |
| POST | `/api/audio/say` | TTS 车端播报 | `{text, lang}` |
| POST | `/api/audio/stop` | 停止播放 | `{}` |
| POST | `/api/audio/record/start` | 开始录音 | `{duration}`（0=手动停） |
| GET | `/api/audio/record/status` | 录音状态 | 无 |
| POST | `/api/audio/record/stop` | 停止录音 | `{}` |
| GET | `/video_feed` | MJPEG 视频（代理 8080） | 无 |
| GET | `/api/nav/stack/status` | 导航栈状态 | 无 |
| POST | `/api/nav/stack/start` | 启动导航栈 | `{wait_ready, timeout}` |
| POST | `/api/nav/stack/stop` | 关闭导航栈 | `{}` |
| GET | `/api/nav/map` | 导航地图元数据 | 无 |
| GET | `/api/nav/map/image` | 导航地图图片（image/jpeg） | 无 |
| POST | `/api/nav/coord/pixel_to_map` | 像素→世界坐标换算 | `{screen_x, screen_y, display_w, display_h}` |
| POST | `/api/nav/initial_pose` | 设置 AMCL 初始位姿 | `{x, y, yaw}` |
| GET | `/api/nav/patrol/route` | 当前巡逻路线 | 无 |
| POST | `/api/nav/patrol/route` | 设置巡逻路线 | `{points: [{x,y}, ...]}` |
| GET | `/api/nav/patrol/status` | 巡逻运行状态 | 无 |
| POST | `/api/nav/patrol/start` | 开始巡逻 | `{}` |
| POST | `/api/nav/patrol/stop` | 停止巡逻 | `{}` |
| POST | `/api/nav/patrol/clear` | 清除巡逻路线 | `{}` |
| POST | `/api/nav/patrol/loop` | 设置循环模式 | `{loop: bool}` |
| GET | `/api/nav/patrol/routes` | 已保存路线名列表 | 无 |
| GET | `/api/nav/patrol/routes/<name>` | 已保存路线详情（含 points） | 无 |

### `/api/status` 返回结构

```json
{
  "code": 0,
  "data": {
    "sensors": {
      "temperature": 25.0, "humidity": 50.0, "smoke": 0, "pm25": 0,
      "pressure": 1013.0, "light": 500, "co2": 400,
      "gps": { "lat": 0, "lon": 0 }
    },
    "battery": 12.0,
    "ip": "192.168.137.23"
  }
}
```

> `battery` 是**电压值**（约 9.0V 空 ~ 12.6V 满），App 侧线性估算成百分比展示。

## 四、底盘移动命令映射（重要，易踩坑）

`/api/move` 与 5001 的 `cmd` 取值**只有以下几种**：

```
forward / backward / left / right / left_shift / right_shift / stop
```

- `left`/`right` = **转向**（原地/差速转）
- `left_shift`/`right_shift` = **麦克纳姆轮平移**
- ⚠️ 文档某些版本提到的 `rotate_left`/`rotate_right` 在 `/api/move` 上**不存在**，勿用

App 内部方向枚举 → REST cmd 的映射约定：

| 内部方向 | REST cmd |
|---------|----------|
| 前进 | `forward` |
| 后退 | `backward` |
| 左平移 | `left_shift` |
| 右平移 | `right_shift` |
| 左转 | `left` |
| 右转 | `right` |
| 刹车/停 | `stop` |

`speed` 范围 10~100；`duration` 为秒（HTTP 用；带 duration 到时自动停，`stop` 时忽略）。

## 五、5001 低延迟控制通道

**目的**：HTTP `/api/move` 每次请求要 TCP 握手 + Flask 路由 + JSON 解析，延迟 10-50ms；
5001 是纯 TCP socket，直发文本行，延迟 ~2-5ms，适合摇杆高频控制。

- 与 5000 **同 IP 同进程**，端口固定 5001
- 协议：每行一条 `<cmd> [speed] [duration]\n`，空格分隔，换行结尾
- **fire-and-forget**，无返回
- `cmd` 取值同上（含 `rotate_left/rotate_right`，5001 支持更全）

示例：
```
forward 50 0.3
left 30 0.5
stop
```

**App 的接入策略**：
1. REST 连接成功后，用同 IP + 5001 额外建一条 TCP 长连接
2. 摇杆/急停**优先走 5001**；若 5001 未连通或 send 失败，**自动回退** HTTP `/api/move`
3. 长连接保活，多条指令复用同一连接省握手开销
4. TCP(6000) 模式下不使用 5001

> 另一端若追求手感一致，建议同样实现"5001 优先 + HTTP 回退"，否则纯 HTTP 摇杆会有明显延迟。

## 六、视频流（统一走 8080）

- 唯一推荐源：**ROS2 `web_video_server`（端口 8080）**
  ```
  http://<小车IP>:8080/stream?topic=/camera/color/image_raw
  ```
- 5000 的 `/video_feed` 本质是代理 8080；本 App 两种模式**都直连 8080** 简化逻辑
- 旧版 6500 `/index2` **已弃用**，勿再用
- 前置条件：车端 Docker 容器 + 视频服务（`v1` 命令）必须起着，否则画面全黑
- 安卓端可用 WebView 加载该 URL，或用支持 MJPEG 的图像组件

## 七、云平台告警 API（端口 8000）

**独立服务器** `http://8.140.28.233:8000`，与小车不同机。同样是 `{code,msg,data}` 包络。

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/healthz` | 健康检查（无需鉴权） |
| GET | `/api/v1/fire-alarms?page=&size=` | 告警列表（分页） |
| GET | `/api/v1/fire-alarms/{id}` | 告警详情（多 `raw_payload`） |

- **鉴权**：可选，若云端开启则请求头带 `Authorization: Bearer <token>`；未开启留空即可
- 告警记录关键字段：`alarm_type`（`confirmed_fire`/`suspected_smoke`/`ai_unavailable`）、
  `occurred_at`、`confidence`(0~1)、`evidence_url`（证据图）、`detection_classes`、`car_id`

⚠️ **明文 HTTP 图片坑**：`evidence_url` 是 `http://8.140.28.233/evidence/...`（非 HTTPS）。
HTTP 数据接口能正常拿到，但**图片组件加载明文 HTTP 可能被系统安全策略拦截**，
需在网络安全配置中显式放行该域名的明文流量（安卓端同理，注意 `usesCleartextTraffic`/
`network_security_config`）。

## 八、页面 ↔ 接口对应关系

按页面组织，说明每个页面对接了什么：

### 1. 连接页（Connect）
- 选择 TCP / REST 模式；填小车 IP、端口
- **TCP 模式**：填 6000 控制端口 + 8080 视频端口 → 建立 6000 长连接
- **REST 模式**：填 5000 REST 端口 + 8080 视频端口 → `GET /api/health` 测通 →
  成功后额外建 5001 控制通道 → 写入全局状态 → 进主页
- 端口配置本地持久化（IP/各端口/后端类型）

### 2. 主页（Home）
- 连接后（REST 模式）**每 3 秒轮询 `GET /api/status`**
- 展示真实：电量（电压估算%）、烟雾值、机柜温度
- 无数据源的指标（如待巡检点）显示 `--`
- 入口按钮：开始巡检 → 地图页；切换手动接管 → 横屏接管页

### 3. 告警页（Alert）
- 进入时 `GET /api/v1/fire-alarms?page=1&size=1` 取**最新一条**云平台告警
- 展示：识别类型、命中类别、置信度、发生时间、车辆、处置建议
- 顶部框加载该告警的 `evidence_url` 证据图（失败/无图回退占位）
- 三态：加载中 / 有告警 / 无告警空态

### 4. 横屏手动接管页（Manual Landscape）
- **视频**：全屏加载 8080 原流；未连接设备显示深灰占位（非白屏）
- **运动摇杆**：拖拽 → 方向+速度 → REST 模式走 5001（回退 HTTP），TCP 模式走 6000 Hex 帧
- **视角摇杆**：右下，云台/视角控制（预留）
- **急停**：REST 走 `stop`（5001/HTTP），TCP 走刹车帧
- **右侧工具列（5 个）**：
  - 灯光 → 仅 6000 有接口，REST 模式占位
  - 语音 → **单击** TTS 喊话（`/api/audio/say`）；**长按** 录音开关（`/api/audio/record`）
  - 拍照 → 仅 6000 有接口，REST 模式占位
  - 雷达 → 本地开关雷达小窗显隐（纯前端）
  - 地图 → 跳转地图导航页
- **布局设置**：右上齿轮进入，可拖拽调整摇杆/雷达位置

### 5. 布局设置页（Manual Settings）
- 横屏，拖拽移动摇杆/雷达/视角摇杆的屏幕位置
- 保存/重置为纯前端行为，不涉及车端接口

### 6. 地图导航页（Map Nav）—— 单点导航
- `GET /api/nav/map` + `/api/nav/map/image` 加载栅格地图
- 点选地图 → `POST /api/nav/coord/pixel_to_map` 换算世界坐标 → `POST /api/nav/initial_pose` 设置起点
- `POST /api/navigate`（旧版，已保留兼容）
- 仅 REST 模式可用；适合单点快速导航，不适合巡检路线编排

### 7. 配送/巡逻页（Delivery）—— 多途经点巡检路线

移动端不做"在地图上逐点点击选坐标"——手机屏幕太小，栅格图上精确定位体验极差。
改为**双模式**，从车端 `/api/nav/patrol/*` 系列接口获取路线数据：

#### 模式 A：选择已保存路线
- `GET /api/nav/patrol/routes` 拉取车端已保存的路线名列表（如 `campus`、`warehouse`）
- 以卡片列表展示，点击选中某条 → `GET /api/nav/patrol/routes/<name>` 获取途经点坐标 → `POST /api/nav/patrol/route` 设为当前路线
- 适合预制好的标准巡检方案

#### 模式 B：自选预设固定点建路线
- 页面内置 6 个预设巡检点（配电柜 A、配电柜 B、仓储通道、消防器材点、充电待命点、应急物资点 D），以 2 列网格展示，每一点带友好名称 + 坐标
- 用户依次点击 → 点加入下方"自选路线"面板（可删、可清空、可查看序号）
- 确认后 `POST /api/nav/patrol/route` 把途经点数组发到车端
- **预设点的坐标值可按实际地图场景替换**，无需改 UI 逻辑

#### 通用控制
- 每 4 秒轮询 `GET /api/nav/patrol/status` + `GET /api/nav/stack/status`
- 导航栈未就绪 → 显示提示 + "启动导航栈"按钮（`POST /api/nav/stack/start`）
- 已就绪且已设置路线 → 显示途经点列表 + "开始巡逻"（`POST /api/nav/patrol/start`）/ "停止巡逻"（`POST .../stop`）
- 巡逻中 → 实时进度条（`navigating_waypoint_N` → 百分比）
- 始终可跳转到地图导航页做单点导航

## 九、关键决策与坑位清单

1. **横竖屏**：手动接管/布局设置是**横屏页**，其余是竖屏。页面切换时主动设置屏幕方向；
   从设置页返回接管页要重新置横屏（否则可能残留竖屏）。
2. **移动命令别用 `rotate_*` 发给 `/api/move`**，只有 5001 支持全集；HTTP 只认那 7 个。
3. **电量是电压不是百分比**，需自行换算。
4. **视频依赖车端 8080 服务起没起**，App 侧无法感知，画面黑先查车端。
5. **证据图是明文 HTTP**，图片组件需放行 cleartext。
6. **5001 是可选加速**，务必实现 HTTP 回退，避免 5001 挂了完全不能动。
7. **灯光/拍照只有 6000 通道**，纯 REST 方案暂不支持，需要就并行开一条 6000 socket。
8. **语音是车端能力**：录音存 wav 在车上，TTS 让车上扬声器发声，都不是手机本地录/放。

## 十、未对接 / 待评估（供另一端规划）

以下车端有能力但本 App **暂未对接**，列出供参考：

| 能力 | 车端接口 | 说明 |
|------|---------|------|
| 语音助手（唤醒/声纹/指令） | ROS2 编排，非简单 REST | 需车端状态机，非手机侧驱动 |
| 人脸识别 | Django 独立服务 :8000（另一套） | 双因子认证场景 |
| 多车协同/编队 | 协调中心 PC :8888 | `/api/formation`、`/api/move_all` 等 |
| 灯光/拍照（REST 下） | 仅 TCP 6000 | 需并行 6000 通道 |
| 巡检报告页 | 暂无对应车端接口 | 目前为静态占位 |
